## 2. UI摩擦ポイント分析

### 2.1 ガントチャートの情報密度とスクロール

| 項目 | 評価 |
|------|------|
| 摩擦レベル | Medium |
| 影響ペルソナ | A, B |
| 関連画面 | SCR-003 |

**問題の詳細**:

設計上、ガントチャートは「呼吸感のある余白」を重視しているが、タスク数が30を超えると縦スクロールが発生し、全体俯瞰が困難になる。バー間余白8px + バー高さ28px = 1行36px であり、30タスクで1,080pxとなる。一般的なデスクトップ画面（高さ900px前後、ヘッダー・タイムラインヘッダー考慮で有効600px程度）では約17タスクまでしか一度に表示できない。

加えて、横方向も月ビューでは十分だが、日ビューで2か月分のタスクを表示する場合は横スクロールも発生し、二次元スクロールが必要になる。

**影響**:
- 「今何をすべきか」が一目でわかるという差別化ポイントが、タスク増加時に損なわれる
- 二次元スクロールはユーザーの空間認識に高い認知負荷をかける

### 2.2 カンバン - ガント間のビュー切替時のコンテキスト維持

| 項目 | 評価 |
|------|------|
| 摩擦レベル | High |
| 影響ペルソナ | B（チームリーダー） |
| 関連画面 | SCR-003, SCR-004 |

**問題の詳細**:

仕様書によると、ビュー切替はタブ（Gantt / Kanban / Calendar / List）で行われる。現在の設計では、ビュー切替時に以下のコンテキストが失われるリスクがある:

1. **スクロール位置**: ガントチャートで特定のタスクを見ていた場合、カンバンに切り替えると最上部に戻る
2. **フィルタ状態**: ガントチャートで「High優先度のみ」にフィルタしていた場合、カンバンでも同じフィルタが適用されるかが不明確
3. **選択中タスク**: ガントでタスクAを選択していた場合、カンバンに切り替えた際にタスクAがハイライトされるべきだが、現仕様では `UIStore.viewMode` の変更のみで選択状態の引き継ぎが定義されていない

**影響**:
- ミーティング中にガントチャートで全体像を確認した後、カンバンで詳細を確認する際に文脈が断絶する
- フィルタの再設定が必要になり、操作ステップが増加する

### 2.3 タスク作成フローのステップ数

| 項目 | 評価 |
|------|------|
| 摩擦レベル | Medium |
| 影響ペルソナ | A, B |
| 関連画面 | SCR-003, SCR-006 |

**問題の詳細**:

仕様書のタスク作成フローを分析すると、最小ステップは以下の通り:

1. 「+ Add Task」ボタンをクリック → タスク作成ダイアログ表示
2. タイトルを入力（必須）
3. ステータスを選択（デフォルト: TODO）
4. 優先度を選択（デフォルト: Medium）
5. 開始日を設定（任意）
6. 終了日を設定（任意）
7. 送信ボタンをクリック

最小3ステップ（ボタンクリック → タイトル入力 → 送信）ではあるが、ガントチャートで意味のあるタスクにするには日付設定が必要であり、実質5~7ステップとなる。Linearのようなキーボード中心のツールでは、`Cmd+K` → タイトル入力 → Enter で即座にタスクが作成できることと比較すると摩擦がある。

**影響**:
- 思いついたタスクを素早くメモする「クイック入力」のユースケースに対応できない
- ダイアログを開く→閉じるの繰り返しが、複数タスクの連続作成時にストレスになる

### 2.4 モバイルでのガントチャート操作性

| 項目 | 評価 |
|------|------|
| 摩擦レベル | High |
| 影響ペルソナ | A（スマートフォン確認時）, B（タブレットミーティング時） |
| 関連画面 | SCR-003 |

**問題の詳細**:

レスポンシブ方針では、モバイル（< md: 768px未満）でガントチャートは「横スクロール」対応とされている。以下の課題がある:

1. **横スクロール + 縦スクロール**: モバイルでの二次元スクロールはタッチ操作と極めて相性が悪い。ユーザーは意図せず斜めにスクロールしてしまう
2. **タッチターゲット**: バー高さは36px（タッチデバイス）で最小要件44pxを下回る。バーをタップして詳細を開く操作で誤タップが発生しやすい
3. **ドラッグ操作**: モバイルでのガントバードラッグは、スクロールとドラッグの区別が困難。dnd-kitの `activationConstraint` を適切に設定しないとスクロールが阻害される
4. **ピンチズーム**: ブラウザのピンチズーム（ページ全体の拡大）とガントチャートのタイムスケール変更ピンチズームが競合する可能性がある

**影響**:
- モバイルでのガントチャートが事実上使い物にならないリスク
- タブレットでのミーティング中の利用体験が著しく低下する

### 2.5 フィルタリングUIの発見可能性

| 項目 | 評価 |
|------|------|
| 摩擦レベル | Medium |
| 影響ペルソナ | B |
| 関連画面 | SCR-003, SCR-004 |

**問題の詳細**:

仕様書では、SCR-003のUI要素一覧に「SCR-003-11: FilterButton - フィルタ表示切替」が定義されているが、以下の懸念がある:

1. **発見可能性**: フィルタボタンがヘッダー右端に配置され、他のアクションボタン（+ Add Task）と並列される。ユーザーがフィルタ機能の存在に気づかない可能性がある
2. **フィルタのAND/OR組み合わせ**: requirements.mdではAND/OR組み合わせフィルタが要件だが、UI上での表現方法が仕様書で定義されていない
3. **フィルタ状態の視認性**: フィルタ適用中であることがユーザーに明示されるUIがない。「なぜタスクが見えないのか」と混乱するリスクがある
4. **フィルタプリセット**: 頻繁に使うフィルタ条件の保存・呼び出し機能がない

**影響**:
- チームリーダーが特定のタグや優先度でフィルタしてミーティングに使う際、毎回フィルタを再設定する手間が生じる
- フィルタ適用中であることを忘れ、タスクが消えたと誤認するインシデントが発生する

### 2.6 タスク詳細Sheetの情報量

| 項目 | 評価 |
|------|------|
| 摩擦レベル | Low-Medium |
| 影響ペルソナ | A, B |
| 関連画面 | SCR-006 |

**問題の詳細**:

タスク詳細Sheet（SCR-006）には以下の要素が含まれる: タイトル、ステータス、優先度、進捗スライダー、開始日、終了日、期限日、タグ、説明（Markdown）、依存関係、コメント、削除ボタン。

これらを右サイドSheetに収めると、特に説明とコメントが長い場合にSheet内のスクロールが発生する。ガントチャート自体のスクロールとSheet内のスクロールが同時に存在し、どちらがスクロールされるかがマウスカーソル位置に依存する不安定さがある。

**影響**:
- 長い説明文やコメント履歴を持つタスクで、Sheet内の操作性が低下する
- 日付ピッカー等のポップオーバーがSheet境界と干渉する可能性がある

## 3. 改善提案

### 3.1 ガントチャートの情報密度対策

| 項目 | 内容 |
|------|------|
| 摩擦ポイント | 2.1 ガントチャートの情報密度とスクロール |
| 優先度 | **P1 (High)** |
| 実装難易度 | Medium |

**改善案**:

1. **折りたたみ可能なグループ化**: プロジェクト内のタスクをタグ、優先度、または手動グループで折りたたみ可能にする。展開/折りたたみでバー表示数を制御し、注目すべきタスクだけを表示できるようにする
2. **ミニマップの拡張適用**: 現仕様ではxl+（1280px以上）でのみミニマップ表示だが、lg（1024px以上）から表示し、ミニマップ上でのナビゲーションを可能にする。ミニマップにビューポート矩形を表示し、ドラッグで移動できるようにする
3. **オートフォーカス「今日」**: ページ読み込み時に今日線が画面中央に来るよう自動スクロールし、期限が近いタスクを上位に表示するソートオプションを提供する

### 3.2 ビュー切替時のコンテキスト維持

| 項目 | 内容 |
|------|------|
| 摩擦ポイント | 2.2 カンバン - ガント間のビュー切替時のコンテキスト維持 |
| 優先度 | **P0 (Critical)** |
| 実装難易度 | Medium |

**改善案**:

1. **共有フィルタ状態**: `UIStore` にビュー横断のフィルタ状態を保持する。ガントで設定したフィルタがカンバン・カレンダー・リストにも自動適用されるようにする

```typescript
// UIStore のフィルタ状態（ビュー横断で共有）
interface SharedFilterState {
  statusFilter: TaskStatus[];
  priorityFilter: Priority[];
  tagFilter: string[];
  dateRangeFilter: { start: Date | null; end: Date | null };
  searchQuery: string;
}
```

2. **選択タスクのハイライト維持**: `TaskStore.selectedTask` がビュー切替後も維持され、切替先のビューでもそのタスクがハイライト表示される（カンバンではカードがハイライト、カレンダーでは該当日がハイライト）
3. **スクロール位置の復元**: 各ビューのスクロール位置を `UIStore` にキャッシュし、ビューを切り戻した際に前回のスクロール位置を復元する

### 3.3 タスクのクイック入力

| 項目 | 内容 |
|------|------|
| 摩擦ポイント | 2.3 タスク作成フローのステップ数 |
| 優先度 | **P1 (High)** |
| 実装難易度 | Low |

**改善案**:

1. **インラインクイック入力**: ガントチャートのタスクリスト末尾に常時表示のテキスト入力欄を配置する。タイトルを入力してEnterキーを押すだけでタスクが作成される（ステータス: TODO、優先度: Medium、日付: 今日〜翌週がデフォルト）。詳細は後からSheetで設定する
2. **キーボードショートカット**: `N` キーでクイック入力にフォーカスする。LinearライクなUXを提供する
3. **カレンダービューからの作成**: 日付セルクリック時、タスク作成ダイアログのstartDateがプリセットされる機能は現仕様にあるが、同様にガントチャートの空き領域をクリックしても日付プリセット付きで作成できるようにする

### 3.4 モバイルのガントチャート代替表示

| 項目 | 内容 |
|------|------|
| 摩擦ポイント | 2.4 モバイルでのガントチャート操作性 |
| 優先度 | **P1 (High)** |
| 実装難易度 | High |

**改善案**:

1. **モバイル向け簡易タイムラインビュー**: 768px未満ではガントチャートを従来のバー表示にせず、縦方向のタイムラインリスト（日付順のカード一覧）に変換する。各カードに進捗バー、優先度バッジ、期限を表示する。これにより二次元スクロールを排除する
2. **タッチターゲットの補正**: タッチデバイスでのバー高さを36pxから44pxに引き上げ、WCAG要件を満たす
3. **ドラッグとスクロールの分離**: dnd-kitの `activationConstraint` に `distance: 10` を設定し、10px以上のドラッグ距離でのみD&Dを発動させる。小さな移動はスクロールとして処理する
4. **ピンチズーム分離**: ガントチャート領域に `touch-action: pan-y` を設定し、横方向のピンチはタイムスケール変更に、縦方向のスクロールはブラウザネイティブに委ねる

### 3.5 フィルタUIの改善

| 項目 | 内容 |
|------|------|
| 摩擦ポイント | 2.5 フィルタリングUIの発見可能性 |
| 優先度 | **P2 (Medium)** |
| 実装難易度 | Medium |

**改善案**:

1. **フィルタバーの常時表示**: フィルタボタンではなく、プロジェクトヘッダー下部にフィルタチップを常時表示する。ステータス、優先度、タグのクイックフィルタをワンクリックで適用可能にする
2. **フィルタ適用インジケーター**: フィルタ適用中は「X件のフィルタ適用中」バッジをフィルタ領域に表示し、「すべてクリア」ボタンを併置する
3. **フィルタプリセット保存**: よく使うフィルタ条件に名前を付けて保存し、ドロップダウンから即座に呼び出せるようにする（localStorageに保存）
4. **AND/OR UIの明確化**: フィルタチップ間に「AND」「OR」のトグルを配置し、組み合わせロジックを視覚的に明示する

### 3.6 タスク詳細Sheetの構造改善

| 項目 | 内容 |
|------|------|
| 摩擦ポイント | 2.6 タスク詳細Sheetの情報量 |
| 優先度 | **P3 (Low)** |
| 実装難易度 | Low |

**改善案**:

1. **セクション折りたたみ**: 説明、依存関係、コメントの各セクションを折りたたみ可能なアコーディオンにする。初期表示ではメタ情報（ステータス、優先度、進捗、日付）のみを展開し、詳細は必要時に展開する
2. **Sheet幅の動的調整**: デスクトップでは400pxをデフォルトとし、説明やコメントの編集時にはSheetを600pxに拡幅するオプションを提供する
3. **タブ分割**: Sheet内を「Details」「Comments」「Activity」の3タブに分割し、1つのスクロール領域あたりの情報量を削減する

### 4.4 カラーコントラスト評価

#### WCAG AA適合判定

| テキスト/背景の組み合わせ | コントラスト比 | 判定 | 備考 |
|-------------------------|---------------|------|------|
| Primary text (`#2d2d2d`) on Background (`#fafaf8`) | 12.3:1 | PASS | 通常テキスト使用可 |
| Dark mode text (`#e4e4e7`) on Dark bg (`#16161a`) | 14.2:1 | PASS | 通常テキスト使用可 |
| Muted text (`#9ca3af`) on Background (`#fafaf8`) | 3.0:1 | FAIL (通常テキスト) | 大テキスト（18px+）のみ使用可 |
| Accent (`#e07a5f`) on Background (`#fafaf8`) | 3.4:1 | FAIL (通常テキスト) | 大テキスト（18px+）のみ使用可。ボタンラベル等は白背景上で使用不可 |
| White on Accent (`#e07a5f`) | 3.1:1 | FAIL (通常テキスト) | 大テキスト（18px+）のみ使用可 |
| Gantt bar label on Terracotta (`#e07a5f`) | ~3.1:1 (白), ~3.8:1 (暗) | 要注意 | ガントバー内ラベル(12px)は通常テキスト扱い。コントラスト不足のリスク |

#### 要対策事項

1. **ガントバー内ラベル**: 12px（0.75rem）は通常テキストに該当するため、コントラスト比4.5:1が必要。テラコッタ(`#e07a5f`)背景上の白テキストは3.1:1でAA未達。対策として、進行中バーのラベルを暗色(`#1a1a2e`)にするか、ラベルをバー外部に移動する
2. **ミュートテキスト**: キャプション（14px）やプレースホルダーにミュートカラーを使用する場合、コントラスト比3.0:1では不足。`#737373` 程度まで暗くすることで4.5:1を確保できる

## 5. 情報アーキテクチャ評価

### 5.1 ナビゲーション構造の妥当性

#### 現在の構造

```
[Sidebar]
```

#### 評価

| 項目 | 評価 | コメント |
|------|------|---------|
| 階層の深さ | 良好 | 最大3階層（Home → Project → View）で適切 |
| 到達ステップ数 | 良好 | どの画面にも最大3クリックで到達可能 |
| プロジェクト切替 | 要改善 | サイドバーにプロジェクト一覧がないため、プロジェクト切替にはHomeに戻る必要がある |
| ビュー切替 | 良好 | タブUIで直感的に切替可能 |
| 通知へのアクセス | 良好 | ヘッダーのベルアイコン + 専用ページ |

#### 改善提案

1. **サイドバーへのプロジェクト一覧追加**: サイドバーに直近アクセスしたプロジェクト（最大5件）を表示し、ワンクリックで切替可能にする。全プロジェクト一覧はHomeに維持する
2. **パンくずリスト**: プロジェクト内の画面にパンくずリストを追加し、現在のコンテキスト（Home > Project A > Gantt）を明示する

### 5.2 画面間の遷移フローの自然さ

#### 主要フロー

```
ログイン → ダッシュボード → プロジェクト選択 → ガントチャート(デフォルト)
                                                ├── カンバン(タブ切替)
                                                ├── カレンダー(タブ切替)
                                                ├── リスト(タブ切替)
                                                ├── タスク詳細(Sheet)
                                                └── 設定(別ページ)
```

#### 評価

| 遷移 | 評価 | コメント |
|------|------|---------|
| ダッシュボード → プロジェクト | 自然 | カードクリックで遷移。戻りはブラウザバック or サイドバーHome |
| ビュー間タブ切替 | 自然 | URLは変わらず（タブ状態のみ変更）。ただしコンテキスト維持が課題 |
| ビュー → タスク詳細Sheet | 自然 | タスクバー/カードクリックでSheetが開く。メインビューは背景に残る |
| タスク詳細 → 依存タスク | やや不自然 | 依存関係リストから依存先タスクへの遷移が未定義。新しいSheetを開くのか、現在のSheetの内容を切り替えるのか |
| 通知 → タスク | 自然 | 通知クリックで該当プロジェクト/タスクへ遷移。Sheetが自動で開く想定 |
| プロジェクト間の切替 | やや不自然 | 一度Homeに戻る必要がある |

### 5.3 情報の階層と優先度

#### ガントチャートビュー（SCR-003）の情報優先度

| 優先度 | 情報 | 表示方法 |
|--------|------|---------|
| 1（最高） | タスクの時間的配置 | ガントバーの水平位置 |
| 2 | タスクの進捗 | バーの塗りつぶし率 + インラインパーセンテージ |
| 3 | 今日の位置 | 赤い点線の今日線 |
| 4 | タスクの依存関係 | 矢印（控えめグレー、ホバーでアクセント） |
| 5 | マイルストーン | ダイヤモンドマーカー |
| 6 | タスクのステータス | バーの色分け |
| 7 | タスクの優先度 | タスク名横のバッジ（必要時のみ） |

**評価**: 情報の優先度は適切に設計されている。バーの色でステータスを、今日線でタイムコンテキストを、矢印で依存関係を表現するレイヤー設計は、認知負荷を段階的に管理できている。ただし、全情報が同時に表示されるため、タスク数増加時は優先度の低い情報（依存矢印、マイルストーン）を非表示にできるトグルが有効。
